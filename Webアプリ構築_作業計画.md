## 1. 計画の目的と前提条件

使用書案に記載された機能のうち、**撮影支援 → 3D再構成 → サーフェス修正 → FEM用形状作成 → 四面体メッシュ生成 → 解析用データ出力**までを実現するWebアプリを段階的に構築するための作業計画を定める。

### 1.1 本計画の前提条件（v0.2で確定）

| # | 前提 | 内容 |
|---|---|---|
| P1 | **静的アプリ** | サーバ側処理を持たない。静的ホスティング（GitHub Pages / Netlify / S3等）で配信し、**すべての処理をブラウザ内（WebAssembly + Web Worker）で実行**する |
| P2 | **GPUなし** | サーバGPUを使わない。端末側もGPU必須としない（表示はWebGLを使うが、計算処理はCPU/WASMで成立させる） |
| P3 | **対応端末** | Android（Chrome）、Windows/Mac PC（Chrome / Edge / Safari）。iOSは動作すれば可の位置づけ |
| P4 | **カメラ入力は汎用** | スマホカメラ / USB接続カメラ / ノートPC内蔵カメラを、ブラウザのカメラAPI（getUserMedia）で同列に扱う。加えて撮影済みの静止画・動画ファイルのアップロード（ファイル選択）にも対応する |
| P5 | **FEM解析は行わない** | 本ツールの成果物は「FEMメッシュと解析用データ」まで。解析（線形静解析・固有値解析等）は外部ソルバ（CalculiX、Abaqus、その他）で行う |

### 1.2 前提条件による使用書案からのスコープ変更

使用書案v0.1のうち、以下は**本ツールの対象外**となる（使用書案の改訂が必要）。

- §19〜§23：材料設定・支持/荷重条件・解析実行・結果表示 → **対象外**。ただし「支持面・荷重面の候補となる面グループの名前付き出力」はオプションとして残す（外部ソルバでの条件設定を楽にするため）
- §13.1：写真的3Dモデル（Gaussian Splatting）→ **対象外**（学習にGPUが必須のため）。撮影確認は点群＋元画像表示で代替
- §24：出力データから解析結果・材料設定・支持/荷重条件を除外。メッシュおよび形状データの出力が中心となる

### 1.3 構築方針

- **パイプラインの背骨（カメラ入力 → 3D再構成 → 水密サーフェス → 四面体メッシュ → 出力）を最短で貫通させる**ことを最優先とする
- 撮影ガイド・品質マップ・面分解などの支援機能は、背骨が通った後に肉付けする
- ブラウザ内CPU処理という制約上、**処理可能な画像枚数・解像度の上限を早期（フェーズ0）に見極め**、それを製品仕様として明記する

## 2. システム構成方針（技術選定）

### 2.1 全体アーキテクチャ（静的アプリ構成）

```text
[静的ホスティング（GitHub Pages / Netlify / S3 等）]
  └ HTML / JS / WASM を配信するだけ（サーバ処理なし）

[ブラウザ（Android / PC）] …… すべての処理がここで完結
  ├ UIスレッド
  │   ├ カメラ入力（getUserMedia：スマホ/USB/内蔵カメラ、ファイルアップロード）
  │   ├ 3Dビューア（WebGL：点群・サーフェス・メッシュ表示）
  │   └ 設定UI（スケール、形状修正、メッシュ設定、出力）
  ├ Web Worker 群（UIを止めずに重処理を実行）
  │   ├ ① フレーム抽出・ブレ判定・特徴点検出
  │   ├ ② SfM（カメラ位置推定）・MVS（密点群生成）〔WASM〕
  │   ├ ③ サーフェス再構成・修復・水密化〔WASM〕
  │   └ ④ 四面体メッシュ生成・品質確認〔WASM〕
  └ ローカル保存（IndexedDB / OPFS）
      └ プロジェクト・段階データ・履歴（エクスポート/インポートはZIP）
```

### 2.2 候補技術スタック

| レイヤ | 第一候補 | 代替候補 | 備考 |
|---|---|---|---|
| フロントエンド | React + TypeScript + Vite | Vue | 静的ビルドしてホスティング |
| 3D表示 | Three.js（WebGL） | Babylon.js | 点群・メッシュ表示。WebGL非対応端末は対象外 |
| カメラ入力 | MediaDevices API（getUserMedia + enumerateDevices） | — | スマホ/USB/内蔵カメラをデバイス選択UIで切替。MediaRecorderで動画、ImageCaptureまたはcanvasで静止画取得 |
| ファイル入力 | File API（静止画一括・動画） | — | デジカメ画像等の取り込み経路 |
| 画像処理 | OpenCV.js（WASM） | 自前実装 | 特徴点検出・マッチング・ブレ判定・マスク処理 |
| SfM（カメラ位置推定） | OpenMVGのWASM移植 | COLMAPのWASM移植 / OpenCV.jsベース自前SfM | **最大の技術リスク。フェーズ0で移植可否と性能を検証** |
| MVS / 密点群 | OpenMVSのWASM移植（低解像度運用） | PatchMatch自前実装 / SfM疎点群＋深度補間 | CPU処理のため解像度・枚数を制限して運用 |
| サーフェス再構成 | PoissonRecon（WASM移植） | ボールピボット法 | 水密サーフェス生成 |
| メッシュ修復・水密化 | Manifold（公式WASMあり） + libigl（WASM） | CGALのWASM移植 | 穴埋め・自己交差解消・ブーリアン |
| 面分解（プリミティブ検出） | RANSAC自前実装（JS/WASM） | CGAL Efficient RANSACのWASM移植 | 平面・円筒・球・円錐 |
| 四面体メッシュ | TetGen（WASM移植実績あり） | Gmsh WASM（実験的）/ fTetWildのWASM移植 | 水密サーフェスからの四面体生成 |
| STEP出力（任意） | opencascade.js（WASM） | 対象外とする | 理想化形状のSTEP化。サイズが大きいため遅延読み込み |
| データ保存 | IndexedDB + OPFS | — | プロジェクト・履歴のローカル保存 |
| プロジェクト入出力 | ZIP（fflate等） | — | 端末間の持ち運び・バックアップ |
| 並列化 | Web Worker + WASM threads（SharedArrayBuffer） | Worker単位の分割のみ | SharedArrayBufferにはCOOP/COEPヘッダが必要。GitHub Pagesはヘッダ設定不可のため coi-serviceworker で対応、またはNetlify等ヘッダ設定可能なホスティングを選ぶ |

> **早期に確定すべき決定事項**
> 1. **SfM/MVSのWASM実装方式**（既存C++のWASM移植か、自前実装か）— フェーズ0の最重要検証項目
> 2. **ホスティング先**（SharedArrayBuffer用ヘッダの設定可否で選定）
> 3. **処理上限の仕様**（画像枚数・解像度・点群数の上限。Android中位機種で現実的な処理時間に収まる値）
> 4. **出力形式の優先順位**（外部ソルバとして何を想定するか：MSH / INP / VTU / STL …）

### 2.3 静的アプリ化に伴う設計上の割り切り

- 動画そのままの処理は重いため、**取り込み時にキーフレーム抽出して静止画列に変換**してから処理する
- MVSはフル解像度では実行せず、**縮小画像での密点群生成**を標準とする（精度への影響はフェーズ0で定量化）
- 処理中はプログレス表示と中断・再開機能を必須とする（スマホでの長時間処理対策）
- 端末のメモリ制約（特にAndroid）を考慮し、段階データはこまめにOPFSへ退避する

## 3. フェーズ分割と作業内容

### フェーズ0：技術検証（PoC）【約3〜4週間】

**ブラウザ内CPU処理でパイプラインが成立するか**を最優先で検証する。UIは作り込まない。

- [ ] 0-1. サンプル対象物の準備（穴付きL型ブラケット等、CAD正解モデルあり。使用書§30準拠）
- [ ] 0-2. カメラ入力の検証：Android Chrome（背面カメラ）、PC＋USBカメラ、ノートPC内蔵カメラで getUserMedia の解像度・フレーム取得を確認
- [ ] 0-3. SfMのWASM化検証：OpenMVG（または代替）をWASMビルドし、20〜60枚程度の画像でカメラ位置推定が完走するか・所要時間を計測
- [ ] 0-4. 密点群生成の検証:縮小画像でのMVS実行、または疎点群ベースの代替手法の品質確認
- [ ] 0-5. PoissonRecon → Manifold → TetGen をWASMで接続し、水密化 → 四面体メッシュ生成まで貫通確認
- [ ] 0-6. スケール設定の検証（既知寸法2点入力による実寸変換の精度）
- [ ] 0-7. 端末別性能計測：Android中位機種 / ノートPCで処理時間・メモリ使用量を計測し、**画像枚数・解像度の上限仕様を決める**
- [ ] 0-8. CAD正解モデルとの寸法・体積比較で到達精度を記録

**完了条件**：ブラウザ内処理のみで「画像一式 → 四面体メッシュ」が貫通し、処理上限仕様と到達精度が文書化されている。

> **フェーズ0の判定**:Android実機での処理が非現実的（例:1プロジェクト1時間超）と判明した場合は、「撮影はスマホ/カメラ・重処理はPCブラウザで実行」という運用（プロジェクトZIPをPCに移して処理）を標準手順とする方針に切り替える。静的アプリ構成のまま対応可能。

### フェーズ1：基盤構築 + パイプライン貫通版【約5〜7週間】

PoCの処理系を静的Webアプリに載せる。撮影ガイドなし・修正機能最小限の「動く縦割り一本」。

#### 1A. 基盤

- [ ] 1A-1. リポジトリ構成・CI・静的ホスティング環境（COOP/COEPヘッダ対応を含む）
- [ ] 1A-2. プロジェクト管理（IndexedDB/OPFS：作成・一覧・設定入力。使用書§7：単位、概寸、スケール設定方法等 ※材料・解析目的の項目は削除）
- [ ] 1A-3. プロジェクトのZIPエクスポート/インポート（端末間移動・バックアップ）
- [ ] 1A-4. Web Worker実行基盤（進捗表示・キャンセル・エラー処理・処理再開）
- [ ] 1A-5. 段階データの保存モデル（使用書§25：撮影→再構成→サーフェス→FEM形状→メッシュを上書きせず履歴保持）

#### 1B. 画像取り込み

- [ ] 1B-1. カメラ撮影UI（getUserMedia：カメラ選択（前面/背面/USB/内蔵）、静止画連続撮影、動画録画）
- [ ] 1B-2. ファイル取り込み（静止画一括・動画。デジカメ画像対応）
- [ ] 1B-3. 動画からのキーフレーム自動抽出（ブレ・重複フレーム除外）
- [ ] 1B-4. 取り込み画像の一覧・不良画像の除外UI

#### 1C. 3D再構成

- [ ] 1C-1. SfMワーカー（特徴点検出 → マッチング → カメラ位置推定 → 疎点群）
- [ ] 1C-2. 密点群生成ワーカー（縮小画像運用）
- [ ] 1C-3. サーフェス生成（Poisson再構成）と水密化
- [ ] 1C-4. スケール設定（2点間実測寸法入力方式を先行実装。使用書§5.2）

#### 1D. 3Dビューア（最小構成）

- [ ] 1D-1. 点群・サーフェス・カメラ位置表示の切り替え（使用書§10.2の一部）
- [ ] 1D-2. 基本操作（回転・移動・ズーム）。Androidタッチ操作対応

#### 1E. 形状クリーニング（最小構成）

- [ ] 1E-1. 机面・背景の自動除去（平面検出 + クラスタリング。使用書§14.1）
- [ ] 1E-2. 手動削除（矩形選択・切断平面。使用書§14.2の一部）
- [ ] 1E-3. 穴埋め・浮遊面除去・水密性チェック（使用書§15.1の一部）

#### 1F. メッシュ生成と出力（最小構成）

- [ ] 1F-1. TetGenによる四面体メッシュ生成（全体要素寸法・要素次数のみ。使用書§18.1の一部）
- [ ] 1F-2. メッシュ品質の基本確認（負体積・アスペクト比・要素数。使用書§18.3の一部）
- [ ] 1F-3. 基本出力（STL / PLY / MSH / VTU / プロジェクトJSON。使用書§24の一部）

**完了条件**：ブラウザだけで「カメラ撮影またはファイル取り込み → 3Dモデル → 四面体メッシュ → ファイル出力」が完結し、出力メッシュが外部ソルバ（CalculiX等）でそのまま読めることを確認済み。

### フェーズ2：撮影支援・品質フィードバック【約4〜6週間】

使用書の中核価値である「撮影不足の検出と追加撮影ガイド」を実装する。

- [ ] 2-1. 撮影中のリアルタイム簡易チェック（ブレ検出・対象物の画面内判定・移動速度警告。使用書§9）※CPU負荷を考慮し低解像度プレビューで実行
- [ ] 2-2. 撮影品質マップ（観測数・視差・再投影誤差に基づく緑/黄/赤/紫/灰の色分け。使用書§10.1）
- [ ] 2-3. 不足箇所選択 → 推奨撮影方向・距離の提示（初期版は粗い方向提示。使用書§10.3, §11）
- [ ] 2-4. 追加撮影データの既存再構成へのマージ（増分再構成）
- [ ] 2-5. 別姿勢撮影（裏返し）と位置合わせ（対応点指定 + ICP。使用書§12）
- [ ] 2-6. 底面の平面補完（切断平面で閉じる・高さ数値指定。使用書§12.3）
- [ ] 2-7. 元画像とモデルの対照表示（写真的3D表示の代替。カメラ位置選択で該当画像を表示）
- [ ] 2-8. 警告メッセージ体系の整備（使用書§27の撮影・形状関係）

**完了条件**：初回撮影 → 品質マップ確認 → 案内に従った追加撮影 → 最終モデル作成、の反復フローが機能する。

- [ ] 3-8. 撮影点群/生サーフェス/修復済み/FEM用形状の重ね合わせ比較（使用書§17.3）

## 5. 主要リスクと対策

| # | リスク | 影響 | 対策 |
|---|---|---|---|
| R1 | **SfM/MVSのWASM化が困難、またはCPU処理が遅すぎる** | 中核機能が成立しない | フェーズ0で最優先検証。画像枚数・解像度の上限仕様で調整。最悪時は「撮影はスマホ・処理はPCブラウザ」運用を標準とする（§3フェーズ0の判定参照） |
| R2 | Android端末のメモリ不足で処理が落ちる | スマホ単独運用が不成立 | 段階データのOPFS退避・処理の分割実行・上限仕様の端末別出し分け。PCでの処理継続経路（ZIP移動）を常設 |
| R3 | 水密化の失敗でメッシュ生成不可 | パイプライン停止 | Manifold等による自動修復 + 手動穴埋めUIを早期実装。フェーズ0でTetGenの入力頑健性を確認 |
| R4 | 3D再構成の精度・頑健性不足（無地・金属・凹部） | 使い物にならない対象が多い | マーカー運用を標準手順化。苦手対象は初期版スコープ外と明記（使用書§3.2） |
| R5 | スケール（実寸）精度の不足 | 出力メッシュが解析に使えない | フェーズ0で誤差を定量化。複数のスケール設定手段（マーカー/既知寸法/2点入力）を提供 |
| R6 | SharedArrayBuffer用ヘッダ（COOP/COEP）の制約 | 並列化不可で処理時間倍増 | ヘッダ設定可能なホスティングを選定、またはcoi-serviceworkerを導入。フェーズ1A-1で確定 |
| R7 | ブラウザ間のカメラAPI差異（特にSafariのMediaRecorder・ImageCapture） | 一部端末で撮影不可 | 静止画連続撮影方式を基本とし動画は補助扱いに。ファイルアップロード経路を常に代替として維持 |
| R8 | ブラウザでの大規模点群・メッシュ表示性能 | 操作性悪化 | LOD・ダウンサンプリング表示。表示用と処理用データを分離 |
| R9 | 出力ファイルが外部ソルバで読めない | ツールの目的不達 | フェーズ1完了条件および4-7で実ソルバ読み込み検証を必須化 |

## 6. 検証方針（受け入れ基準）

使用書§30・§31に基づき、各フェーズ完了時に以下を確認する（解析検証は外部ソルバで実施）。

1. **形状精度**：CAD正解モデルとの比較で、全幅・高さ・奥行き・穴径・体積の誤差を記録（目標値はフェーズ0の結果を踏まえて設定）
2. **メッシュ健全性**：外部ソルバでエラーなく読み込め、解析が実行できること
3. **作業時間**：撮影開始からメッシュ出力までの所要時間（端末別）、うち手動修正時間
4. **再現性**：同一対象物を別日に撮影して同等の結果が得られること
5. **端末網羅**:Android実機・PC＋USBカメラ・ノートPC内蔵カメラの3経路すべてで1〜4を確認

---

## 7. 直近のアクション（フェーズ0着手）

1. テスト対象物（穴付きL型ブラケット等、CADデータのあるもの）を用意する
2. 検証用端末を確保する（Android中位機種、USBカメラ付きPC、ノートPC）
3. OpenMVG（または代替SfM）のWASMビルドを試行し、移植可否を早期に見極める
4. PoissonRecon・Manifold・TetGenのWASMビルドを準備する
5. 各カメラ（スマホ/USB/内蔵）で使用書§8の手順に準じた検証用画像セットを収集する
6. 0-3〜0-5のパイプラインをブラウザ上で貫通させ、精度・処理時間・メモリを記録する
