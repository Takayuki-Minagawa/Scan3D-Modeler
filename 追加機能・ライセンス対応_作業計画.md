## 1. 背景と本計画の範囲

コードベースを調査した結果、以下の2系統の作業を計画する。

1. **ライセンス対応(最優先)** — リポジトリはパブリック公開済みだが LICENSE ファイルがなく、README も「ライセンス未設定」と明記している状態。また、フェーズ0以降で採用予定の外部ライブラリに **AGPL/GPL(強コピーレフト)** のものが含まれており、寛容ライセンスでの公開方針と両立しないため、技術選定の差し替えが必要
2. **追加機能** — PROGRESS.md の既知課題およびコード調査から見つかった、有効性の高い機能追加・改善

## 2. ライセンス対応(A系列: 最優先)

### 2.1 現状

- リポジトリ直下・app/ 配下ともに **LICENSE ファイルなし**。`app/package.json` にも `license` フィールドなし
- README には日英で「ライセンスファイル未収録。条件は明示されていない」と記載済み(誠実だが、ライセンス無指定 = **デフォルトで全権利留保**であり、公開リポジトリとしてはフォーク・利用の法的根拠が曖昧なまま)
- GitHub Pages への自動デプロイ(`.github/workflows/deploy-pages.yml`)が有効 = **ビルド成果物(バンドル)の再配布**が既に発生している

| パッケージ | 用途 | ライセンス | バンドルへの同梱 |

## 3. 将来WASMパイプラインのライセンス適合(B系列: フェーズ0着手前に確定)

### 3.1 問題点

[Webアプリ構築_作業計画.md](Webアプリ構築_作業計画.md) §2.2 の候補技術スタックには、**パブリックリポジトリを寛容ライセンス(MIT等)で公開する方針と両立しないライブラリ**が含まれる。

> 補足(正確な整理): AGPL/GPL のライブラリを使うこと自体は「公開不可」ではないが、WASMとして同梱・配布すると**アプリ全体を AGPL/GPL で公開する義務**が生じる。リポジトリを MIT 等で公開する前提(§2.3 A-1)を採る場合、これらは**同梱できない**ため差し替える。

### 3.2 候補ライブラリのライセンス調査結果と差し替え方針

| 用途 | 現計画の候補 | ライセンス | 判定 | 差し替え/対応 |
|---|---|---|---|---|
| 四面体メッシュ | **TetGen** | **AGPL-3.0**(商用ライセンスはWIASから有償) | ❌ 同梱不可 | **fTetWild(MPL-2.0)へ第一候補を変更**。fTetWildは非水密・自己交差入力に頑健で、Manifold前処理の負荷軽減という技術的利点もある。WASM移植の難度・処理時間はフェーズ0で検証(TetGenより重い点に注意)。次点: TetWild(MPL-2.0) |
| MVS(密点群) | **OpenMVS** | **AGPL-3.0** | ❌ 同梱不可 | 現計画の代替候補どおり **PatchMatch自前実装、またはSfM疎点群+深度補間**へ一本化。COLMAP(BSD-3相当のカスタム表記)のMVSはCUDA前提のためWASM不可。フェーズ0の検証対象を「OpenMVSのWASM化」から「自前MVSの品質確認」に変更 |
| 面分解RANSAC | CGAL Efficient RANSAC(代替候補) | **GPL**(CGALはGPL/LGPL混在。Shape Detection パッケージはGPL) | ❌ 同梱不可 | 第一候補どおり **RANSAC自前実装(JS/WASM)** で確定し、CGALは代替候補から削除 |
| メッシュ修復 | libigl | コアは MPL-2.0 だが **`igl/copyleft/` 配下(CGAL・CORK連携等)は GPL** | ⚠️ 条件付き | **コア(MPL-2.0)のみ使用**と計画に明記。copyleftディレクトリのアルゴリズムは使わない |
| 四面体メッシュ代替 | Gmsh | **GPL-2.0以降** | ❌ 同梱不可 | 代替候補から削除 |
| SfM | OpenMVG | MPL-2.0 | ✅ 可 | そのまま(MPLファイルの改変部分のみソース開示義務。WASM化パッチの公開で満たせる) |
| SfM代替 | COLMAP | BSD系(カスタム表記) | ✅ 可 | そのまま(ただしMVS部は上記のとおり不可) |
| サーフェス再構成 | PoissonRecon(mkazhdan) | MIT | ✅ 可 | そのまま |
| 水密化・ブーリアン | Manifold | Apache-2.0 | ✅ 可 | そのまま |
| 画像処理 | OpenCV.js | Apache-2.0 | ✅ 可 | そのまま |
| COOP/COEP対策 | coi-serviceworker | MIT | ✅ 可 | そのまま |
| STEP出力(任意) | opencascade.js | LGPL-2.1(例外条項付き) | ⚠️ 要検討 | LGPLはWASM静的リンク時の「再リンク可能性」解釈に議論があるため、**採用時に再評価**。任意項目のため当面は対象外のまま |

## 4. 追加機能(C系列: コード調査に基づく提案)

フェーズ0(WASM検証)が最優先である方針は維持しつつ、**フェーズ0と独立に進められ、公開アプリとしての完成度・信頼性を上げる**ものを優先度順に挙げる。C-1〜C-3 は PROGRESS.md の既知課題の解消でもある。

### 優先度: 高

| # | 機能 | 内容・根拠 | 規模 |
|---|---|---|---|
| C-1 | **ストレージ容量の警告と永続化要求** | `navigator.storage.estimate()` で使用量/上限を取得しヘッダー等に表示、閾値超過(例: 80%)で警告。あわせて `navigator.storage.persist()` を要求し、ブラウザ都合のデータ削除(eviction)を抑止。撮影データを失うリスクへの対策として公開アプリでは重要(既知課題に記載済み) | 小 |
| C-2 | **ギャラリーのサムネイル保存** | `capture/Gallery.tsx` は全画像をフル解像度Blob URLで表示しており、動画フレーム数百枚で破綻し得る(既知課題)。取込・抽出時に縮小サムネイル(例: 長辺256px JPEG)を `assets` に併存保存し、一覧はサムネイル、詳細表示のみ原画を読む。ブレ判定ワーカーが既にcanvas縮小処理を持つため流用可能 | 中 |
| C-3 | **1C-4 スケール設定UI(2点間距離)** | 既存計画項目かつ「次にやること#2」。ビューア(`viewer/threeView.ts`)にレイキャストで点群上の2点を選択→実測距離入力→プロジェクト設定にスケール係数を保存。デモデータでもUI・保存経路の実装と検証が可能で、フェーズ0完了を待つ必要がない。副産物として**2点間の距離計測機能**にもなる | 中 |
| C-4 | **coi-serviceworker の導入** | GitHub Pages はCOOP/COEPヘッダを設定できないため、将来のWASMスレッド(SharedArrayBuffer)が現デプロイ構成では動かない。`coi-serviceworker`(MIT)を先行導入し、`crossOriginIsolated` の状態を診断表示すれば、フェーズ0の成果物をそのままPages上で検証できる。開発サーバ(`vite.config.ts`)には設定済みで、本番側だけが未対応 | 小 |

### 優先度: 中

| # | 機能 | 内容・根拠 | 規模 |
|---|---|---|---|
| C-5 | **PWA化(オフライン対応)** | 完全静的・全処理ブラウザ内という構成はPWAと相性がよい。manifest + Service Worker(C-4と統合)でインストール可能にし、電波のない撮影現場でも利用可能に。favicon.svgは作成済みでアイコン流用可 | 中 |
| C-6 | **EXIF読み取りと表示** | 取込画像のEXIF(焦点距離・撮像素子情報・撮影日時)を取得して一覧に表示し、アセットに保存。SfMの内部パラメータ初期値(focalPx)として将来直接使えるため、フェーズ0への布石にもなる。軽量な自前パーサまたはMITライセンスの exifr 等 | 中 |
| C-7 | **バンドルのコード分割** | 748KB(Three.js が主因、既知課題)。ビューア(`viewer/`)を `React.lazy` + dynamic import で分割し、初期表示(プロジェクト一覧)を軽量化。低速回線のスマホでの初回ロード改善 | 小 |
| C-8 | **IndexedDB使用量の内訳表示と個別削除** | プロジェクトごとの概算サイズ(アセット合計)を一覧に表示し、容量圧迫時にどれを消す/ZIP退避するか判断できるようにする。C-1の警告からの自然な導線 | 中 |

### 優先度: 低(フェーズ0以降と要調整)

| # | 機能 | 内容・根拠 | 規模 |
|---|---|---|---|
| C-9 | ZIPエクスポートのストリーミング化 | fflate のストリームAPIで逐次書き出し(既知課題)。`showSaveFilePicker` が使える環境ではディスク直書き。大容量プロジェクトのメモリ不足対策 | 中 |
| C-10 | 段階データのOPFS退避 | 計画§2.3に記載済み・未実装。大きなBlobをIndexedDBからOPFSへ移し、メモリ・性能を改善。DBスキーマ変更を伴うためフェーズ0の成果物サイズ感が見えてから設計 | 大 |
| C-11 | プロジェクト複製 | 設定違いの試行(将来のメッシュ設定比較)に有用。ZIPエクスポート→インポートのID振り直しロジックを内部転用すれば実装は薄い | 小 |
| C-12 | ビューアのスクリーンショット保存 | `preserveDrawingBuffer` または `toBlob` で現在表示をPNG保存。撮影対象の記録・報告用途。フェーズ4の品質レポート(4-6)の先行部品 | 小 |

## 7. 実施結果（2026-07-18）

M1〜M5およびM7（A-1〜A-6、B-1〜B-4、C-1〜C-8）を実装した。M6は、ライセンス適合後の候補スタックを固定し、`scripts/check-phase0-toolchain.sh` と `docs/phase0/` による再現可能な環境診断まで着手した。現環境にはCMake/Emscriptenがないため、外部ツールを暗黙に導入せず、0-3〜0-5の実WASMビルドは次作業として明示している。

低優先のC-9〜C-12は、表に記載したとおりフェーズ0以降に成果物サイズ・端末メモリ・品質レポート要件と合わせて再調整する項目であり、今回の実施マイルストーンには含めない。実装内容と検証結果の詳細は [PROGRESS.md](PROGRESS.md) を正本とする。
